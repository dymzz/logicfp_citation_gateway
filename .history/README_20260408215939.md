# logicfp-citation-gateway

# 🛑 P3 引用网关：使用vibecoding写的 极其死板，毫无优雅可言的项目

如果你在寻找一个能“智能理解”大模型意图、“优雅包容”各种输出格式的 AI 护栏，请立刻关闭这个仓库。

本项目（P3：通用引用完整性网关）是对现代 AI 开发惯例的一次彻底反叛。它没有任何大模型时代的“优雅”，只有上个时代最原始、最教条、最死板的硬逻辑。

## 🧱 毫无弹性的“反智能”设计

在这个系统里，大模型不是无所不能的 AI，而是一个随时可能撒谎的嫌疑人。P3 就是那个只会照本宣科、毫无通融余地的门卫：

* **绝对拒绝“AI 魔法”：** 系统明文规定，严禁使用 Embedding 算相似度，严禁调用 LLM 辅助判断，严禁任何形式的模糊匹配（Fuzzy Match）。
* **粗暴的正则解析：** 没有任何智能语义提取，全靠写死的正则表达式去抠 `[ref:xxx]` 这种固定格式。大模型如果多敲了一个空格，或者换了种引用说法，系统不会去“猜”它的意图，而是直接无脑抛出解析失败（`parse_failed`）。
* **不留情面的硬阻断：** 一旦 P3 在字典（`ContextPool`）里查不到大模型输出的 ID，或者发现明确的池外越界（`outside_pool`）和伪造（`fabricated`），它会直接触发硬拦截（Block）。哪怕语义检查模块（P2）觉得大模型回答得极其完美，P3 也会一刀切地将结果毙掉，绝对的专断独行。
* **严禁“自作聪明”的容错：** 别人家的网关会自动帮你修复引用、补全 ID。而本系统明确把“自动修复”、“自动补全”列为设计红线，错了一点就直接报错。

## ⚠️ 试验性工程，生产环境请谨慎

> **"它毫无优雅可言，因为它的目标是在极端情况下强行拉下电闸。"**

必须强调，本项目是一套**试验性极强、容错率极低**的防线。

* **当前状态：** 本项目目前仅处于“v0.2 接口冻结版”的设计推进阶段。
* **部署警告：** 这套毫无“优雅”可言的死板逻辑，极其依赖宿主系统提供完美的数据字典（`ContextPool`）以及大模型输出完美的格式。在您的上游链路尚未完全稳定之前，贸然将其接入生产环境，这种“宁可错杀一千，绝不放过一个”的逻辑，极大概率会导致大面积的误杀和业务阻断。
* **建议：** 仅建议在离线测试和对合规要求极度严苛、且愿意牺牲大量可用性的严肃场景下进行试验。
