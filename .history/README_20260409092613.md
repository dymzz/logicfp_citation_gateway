## 为什么这不是一套“规则很多的 guardrails”

这个项目的目标，不是把大模型“管得更死”，而是把 **LLM 输出校验** 做成一套 **可解释、可回归、可集成** 的工程协议。

在实际系统里，LLM 的问题通常不只是一句“它答错了”，而是更具体的几类：

- 回答是否真的被当前上下文支持
- 引用是否真的来自当前上下文池
- 为什么这次应该放行、警告、降级或拦截
- 升级后规则是否发生漂移
- 出错后能否稳定复盘，而不是靠人工猜

为了解决这些问题，这个项目把校验链拆成了清晰分层：

- **P2：语义支持判断**  
  判断回答是否被当前上下文语义支持

- **P3：引用完整性判断**  
  判断回答中的引用是否真的属于当前 `ContextPool`

- **主程序聚合层**  
  统一输出 `allow / warn / fallback / block` 等决策结果

- **P4：外部业务插件层**  
  承接行业规则、人工审核、自动纠错、企业流程等能力

这样的设计不是为了追求“抽象好看”，而是为了得到几个直接的工程收益：

### 1. 低歧义

每一层只回答一个问题：

- P2 不负责判断引用归属
- P3 不负责判断语义支持
- P4 不污染通用内核
- 最终决策只由主程序统一生成

这样系统行为更稳定，不会在不同模块里重复做相似判断，导致边界混乱。

### 2. 可审计

当系统输出 `warn` 或 `block` 时，可以明确回答：

- 是语义支持不足
- 还是引用越界
- 还是伪造引用
- 还是业务插件层触发了额外规则

这让错误不再是“模型不太对劲”，而是可以落到具体层次和具体原因。

### 3. 可回归

项目内置了 benchmark、样本模板、回归执行规范和版本控制约束。  
这意味着每次修改 parser、规则或聚合逻辑后，都可以回答：

- 哪些样本继续通过
- 哪些边界发生了漂移
- 是状态变了，还是细节字段变了
- 这次改动是否破坏了历史行为

这对长期维护比“凭感觉调规则”重要得多。

### 4. 可集成

宿主系统不需要理解全部内部细节，只需要认稳定的输入输出协议：

- 输入：`HostP3Request`
- 输出：`P3Result`
- 聚合后输出：`DecisionResult`

这让它更像一套 **可复用的中间层协议**，而不是某个业务里写死的一堆 if-else。

## 这套设计的真正价值

这套设计的价值，不在于“限制更多”，而在于：

> **即使接入方不认同这套设计哲学，也能直接获得它的工程收益。**

也就是说，别人不需要先理解某种架构信仰，  
只需要看到结果：

- 更容易解释为什么拦截
- 更容易复盘为什么失败
- 更容易验证升级没有漂
- 更容易把通用校验和行业规则拆开

这也是它真正有机会被复用、被传播、甚至形成事实标准的原因。

---

## 它适合解决什么问题

这套体系适合用于：

- 企业知识库问答
- RAG 输出校验
- 带引用的总结/报告生成
- 对可审计性要求高的内部 AI 系统
- 需要把“通用校验”和“业务规则”分层管理的 LLM 应用

它不是为了替代行业专家系统，  
而是为了给 LLM 系统提供一个稳定、透明、可验证的校验底座。

## 架构总览（P2 / P3 / P4）

这个项目把 LLM 输出校验拆成三层能力，再由主程序统一聚合决策。

```text
LLM Output
    ↓
P2: Semantic Support
    ↓
P3: Reference Integrity
    ↓
Main Decision Aggregator
    ↓
P4: Business / Domain Extensions
    ↓
Final Decision / Audit / Workflow
```

### P2：语义支持判断层

P2 负责回答：

- 当前回答是否被给定上下文语义支持
- 是否存在明显的语义漂移、过度概括或不受支持的结论
- 在“内容层”上，这个回答是否值得继续进入后续放行链

P2 不负责：

- 判断引用是否真的来自当前上下文池
- 判断引用 ID 是否存在
- 处理业务审批流、行业规则、人工审核策略

一句话理解：

P2 解决“这段回答在语义上是否站得住”。

### P3：引用完整性判断层

P3 负责回答：

- 回答中的引用是否可解析
- 引用是否能绑定到当前 ContextPool
- 是否存在池外引用、伪造引用或无法归属的引用

P3 的核心原则是：

- 只基于当前 ContextPool
- 不访问外部系统补全引用
- 不调用 embedding / LLM 做模糊猜测
- 不自动修复输出
- 不直接输出最终业务 verdict

一句话理解：

P3 解决“这段回答里的引用是否合法属于当前上下文池”。

### 主程序聚合层

主程序聚合层负责把 P2 和 P3 的结果收敛成统一决策。

它负责：

- 合并 P2Result 与 P3Result
- 生成统一的 DecisionResult
- 输出 allow / warn / fallback / block
- 统一映射 trigger、reason、error code
- 记录审计事件与运行结果

这里的关键点是：

- P2 与 P3 都不是最终决策器
- 最终决策只由主程序统一生成
- P3 的 hard-fail 不能被 P2 覆盖

一句话理解：

聚合层解决“系统最终应该怎么处理这次输出”。

### P4：业务 / 领域扩展层

P4 不是通用内核的一部分，而是外部可选扩展层。

它适合承接：

- 行业专项规则
- 人工审核流程
- 自动纠错 / 重试
- 企业审批流
- 合规策略
- 与外部平台或内部系统的业务集成

这样做的原因是：

- 让通用校验内核保持稳定
- 不把某个行业的业务规则写死进 P2 / P3
- 让不同业务可以在不破坏核心 contract 的情况下自由扩展

一句话理解：

P4 解决“不同业务场景如何在通用校验之上加自己的规则和流程”。

设计原则

这套架构有几个核心原则：

1. 单层单责
- P2 只做语义支持判断
- P3 只做引用完整性判断
- 主程序只做统一决策聚合
- P4 只做业务扩展
2. 核心保守，外层灵活

通用内核尽量少变：

输入输出 contract 稳定
状态枚举稳定
trigger / error code 稳定
benchmark 与回归口径稳定

业务差异尽量放到外层：

审核流
合规策略
行业规则
自动纠错
企业集成
3. 先可解释，再可扩展

相比“更聪明地猜”，这套设计更强调：

为什么拦截
为什么告警
为什么进入 fallback
哪一层出了问题
升级后哪条边界发生了漂移
它适合解决什么问题

这套架构更适合以下类型的系统：

企业知识库问答
RAG 输出校验
带引用的总结 / 报告生成
对可审计性要求高的内部 AI 系统
需要把通用校验和业务规则分层管理的 LLM 应用

它不是为了替代行业专家系统，
而是为了提供一个稳定、透明、可验证的校验底座。

实验性声明

Experimental / Work in Progress

本项目目前仍处于实验性、未完成阶段。

当前仓库中的设计、代码结构、benchmark、回归流程和接口定义，主要用于：

验证架构方向
固化核心 contract
迭代 benchmark 与回归能力
评估 P2 / P3 / P4 分层是否稳定可行

这意味着：

部分模块仍在持续演进
接口与行为细节未来仍可能调整
benchmark 覆盖面仍在扩展
当前实现尚未经过完整的生产级验证
请勿直接用于生产环境

在正式完成以下工作之前，不建议直接用于生产环境：

接口进一步冻结
完整 benchmark 与回归体系稳定
错误处理与审计链补全
版本演进与兼容性策略成熟
业务扩展层（P4）经过充分验证

如果你正在评估这个项目，当前更合适的使用方式是：

作为实验性原型
作为设计参考
作为内部 PoC
作为 LLM 校验中间层的研究与迭代基础

而不是直接作为生产级依赖接入关键业务系统。
