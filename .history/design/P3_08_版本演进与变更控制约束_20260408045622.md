第 X 章：P3 版本演进与变更控制约束
X.1 文档目的

本章用于约束 P3 在后续迭代中的版本演进方式与变更控制方式，防止以下问题出现：

parser 一次“小优化”偷偷改变了状态边界
resolver 为了“更智能”突破 ContextPool 边界
benchmark 样本被静默修改，导致历史结果失真
主程序接入方在没有显式版本升级的情况下被迫适配新口径
P3 从“引用归属网关”逐步漂移成“半语义判断器”

本章的目标不是阻止演进，而是确保：

P3 的任何演进都必须是显式的、可追踪的、可回归验证的。

X.2 版本控制的核心原则（冻结）
X.2.1 边界稳定优先于功能扩张

P3 的首要目标不是“支持更多花样引用”，而是保持其核心定义长期稳定：

P3 只判断引用是否属于当前 ContextPool。

任何功能扩张，只要触碰这个定义，都必须被视为高风险改动。

因此，P3 的演进原则必须是：

先保护边界
再扩展能力
最后优化实现

而不能反过来。

X.2.2 公开 contract 稳定优先于内部实现自由

P3 允许内部实现调整，例如：

parser 重构
resolver 优化
evaluator 代码整理
runner 性能改进

但这些调整不得破坏已冻结的公开 contract，包括：

HostP3Request
P3Result
reference_integrity_status 枚举
trigger code 公开集合
主程序接入顺序与聚合边界

内部可以重构，公开 contract 不可随意漂移。

X.2.3 显式变更优先于隐式漂移

P3 不允许“边改边用，改了再说”的演进方式。
任何影响以下内容的改动，都必须被视为显式变更：

状态边界
输出字段含义
trigger 语义
benchmark 真值
主程序集成口径

换句话说：

没有明确变更说明的行为，不得改变 P3 的对外语义。

X.3 版本分层（建议冻结）

建议把 P3 的版本划分为三个层次：

X.3.1 设计版本

用于标识设计口径的冻结状态，例如：

P3 Design Version: v1

设计版本用于描述：

边界定义
contract 定义
benchmark 口径
变更规则

只要这些内容发生正式升级，就应提升设计版本。

X.3.2 实现版本

用于标识代码实现版本，例如：

P3 Implementation Version: 0.3.1

实现版本可以随着代码迭代更频繁变化，用于区分：

bug 修复
小幅重构
性能优化
非破坏性增强
X.3.3 benchmark 版本

用于标识 benchmark 样本集版本，例如：

P3 Benchmark Version: v1

这个版本只在样本集合本身发生显式变化时更新，例如：

新增一组正式样本
删除错误样本
正式更正某个 case 真值
样本组织方式发生升级
X.4 变更类型分级（冻结）

为了避免所有修改都混在一起，本章将 P3 变更分为四类。

X.4.1 A 类：文案级 / 非语义变更

包括但不限于：

注释优化
内部变量重命名
reasons 文案润色
报告模板文字优化
非公开内部日志调整

A 类变更不应影响：

reference_integrity_status
关键集合字段
主程序集成口径
benchmark 真值

A 类变更通常不需要升级设计版本，也不应改 benchmark expected 的核心字段。

X.4.2 B 类：实现级 / 非破坏性变更

包括但不限于：

parser 内部实现重构
resolver 性能优化
evaluator 代码整理
新增非阻塞 metrics
新增更清晰的辅助 triggers（在不改变主语义前提下）

B 类变更允许升级实现版本，但前提是：

benchmark 主状态不漂移
contract 不破坏
主程序集成不受影响

B 类变更必须跑完整 benchmark，并确认无 hard fail。

X.4.3 C 类：边界级 / 语义级变更

包括但不限于：

新增新的引用解析规则并可能改变样本判定
调整 unresolved / partial 的边界
调整 fabricated 判定逻辑
修改 trigger 的公开语义
修改 benchmark 真值

C 类变更属于高风险变更。
这类变更必须：

显式写入变更说明
明确影响范围
跑完整 benchmark
对比旧版报告
说明是否需要升级设计版本或 benchmark 版本

未履行这些要求时，不应接受 C 类变更。

X.4.4 D 类：Contract 级 / 破坏性变更

包括但不限于：

修改 HostP3Request 必需字段
修改 P3Result 核心字段含义
修改冻结状态枚举
移除公开 trigger code
改变宿主调用方式
改变主程序集成优先级

D 类变更必须视为 破坏性升级。
默认不应在当前设计版本内发生。

若确实需要发生，则必须：

升级设计版本
升级实现主版本
给出迁移说明
给出旧版兼容策略或明确声明不兼容
更新 benchmark 与宿主接入文档
X.5 什么情况下必须升级版本
X.5.1 必须升级实现版本的情况

包括：

任何代码级逻辑改动
parser / resolver / evaluator 行为改动
benchmark runner 行为改动
报告生成逻辑改动

只要实现变了，就应升级实现版本。

X.5.2 必须升级 benchmark 版本的情况

包括：

正式新增 benchmark case
删除已有正式 case
调整某条 case 的 expected
调整 manifest 结构
调整 benchmark 分类组织规则

只要样本集本身变了，就应升级 benchmark 版本。

X.5.3 必须升级设计版本的情况

包括：

修改冻结 contract
修改状态枚举
修改 P3 与主程序的职责边界
修改 P3 与 P2 的边界
修改核心禁止项
修改状态判定语义

只要设计口径变了，就不能只改代码或样本，必须升级设计版本。

X.6 变更提交流程建议

建议所有影响 P3 的正式变更都遵循以下流程：

提出变更意图
    ↓
判定变更类型（A / B / C / D）
    ↓
更新版本号
    ↓
修改实现或文档
    ↓
执行完整 benchmark
    ↓
输出对比报告
    ↓
人工审查是否接受

这个流程的重点不在形式，而在于：

每次改动都必须被分类
每次改动都必须被验证
每次改动都必须留下可追踪证据
X.7 变更说明模板（建议冻结）

每次正式变更，建议至少附带以下信息。

X.7.1 最小变更说明字段
change_id:
change_type: A / B / C / D
design_version:
implementation_version:
benchmark_version:
summary:
affected_scope:
expected_impact:
benchmark_result:
breaking_change: yes / no
migration_required: yes / no
X.7.2 示例
change_id: p3-change-2026-001
change_type: C
design_version: v1
implementation_version: 0.4.0
benchmark_version: v2
summary: refine fabricated detection rule for explicit fake-prefixed references
affected_scope: evaluator, fabricated benchmark cases
expected_impact: fabricated cases may change trigger details, status should remain stable
benchmark_result: no hard fail, 2 soft fail
breaking_change: no
migration_required: no
X.8 Benchmark 与变更控制的关系

P3 的 benchmark 不只是测试集，更是变更控制工具。
任何 P3 变更，都必须借助 benchmark 回答两个问题：

这次改动是否改变了核心边界
这次改动是否破坏了历史冻结样本

因此，本章明确规定：

没有 benchmark 对比报告的 P3 变更，不应视为完整变更。

尤其是 C 类和 D 类变更，不能只靠“逻辑上应该没问题”来放过。

X.9 不允许的变更控制做法

这部分是本章红线。

X.9.1 禁止静默修改 benchmark 真值

当代码跑不过时，不能先改 expected 再说。
必须先判断：

代码是不是错了
样本是不是错了
设计是不是正式变了

没有明确理由，不得修改真值。

X.9.2 禁止未分类变更直接合入

任何正式变更都必须先判定属于 A / B / C / D 中哪一类。
“先改了再看影响”这种做法，会让 P3 的版本语义迅速失控。

X.9.3 禁止用实现升级掩盖设计升级

如果某次改动实际上改变了：

状态边界
contract 含义
宿主接入方式

就不能只升实现版本。
这类改动本质上是设计升级，必须显式记录。

X.9.4 禁止绕过 benchmark 直接声称“没问题”

只要改动影响 parser、resolver、evaluator 或主程序集成边界，就必须跑 benchmark。
不能因为改动“看起来很小”就跳过。

X.9.5 禁止破坏性变更无迁移说明

如果发生 D 类变更，就不能只改代码与文档标题。
必须明确告诉接入方：

哪些字段变了
哪些行为变了
原接入方式是否还能工作
是否需要改 benchmark / 聚合器 / 宿主代码
X.10 兼容性原则（建议冻结）

建议 P3 默认遵循以下兼容性原则：

X.10.1 向后兼容优先

在不引入明显设计污染的前提下，优先保持：

公开 contract 不变
枚举不变
主程序集成方式不变
benchmark 主状态不变
X.10.2 新能力默认增量引入

例如新增引用语法支持时，应优先：

新增解析能力
不改变旧 case 判定
不破坏旧 contract

而不是为了支持新语法，把旧逻辑全部推翻。

X.10.3 破坏性升级必须显式宣布

只要不兼容，就必须明确写出来，而不能让接入方在运行时报错时才发现。

X.11 建议的版本记录方式

建议在 P3 目录中保留最小版本记录文件，例如：

p3_versioning.md

用于记录：

当前 design version
当前 implementation version
当前 benchmark version
最近一次正式变更摘要
历史版本里程碑

这样做的目的不是形式化，而是让后续任何人都能快速回答：

P3 现在到底冻结到了什么程度，最近改了什么。

X.12 本章结论

P3 一旦进入正式工程化阶段，最大风险往往不是“没有功能”，而是：

边界在多次小改中被慢慢改没了。

因此，本章冻结三条核心原则：

所有变更必须分类。
所有重要变更必须有 benchmark 对比证据。
所有边界变化必须显式升级版本，而不能隐式漂移。

只要这三条守住，P3 即使持续迭代，也能保持其作为“通用引用完整性网关”的结构纯度。
