第 X 章：P3 回归执行规范与结果报告模板
X.1 文档目的

本章用于定义 P3 benchmark 的执行方式、结果判定方式与报告输出方式，目标不是追求“跑完就算”，而是保证：

benchmark 可重复执行
回归结果可审计
失败原因可定位
历史版本可比较
规则迭代不会因为报告口径不一致而失真

本章的核心目标是把 P3 的 benchmark 从“样本集合”推进为“标准化回归流程”。

X.2 适用范围

本章适用于：

P3 benchmark 的日常执行
解析器、resolver、evaluator 修改后的回归验证
冻结版本之间的结果比较
benchmark 报告归档与复盘

本章不适用于：

P2 语义评测执行规范
主程序完整集成压测
行业专项插件评测流程
模型质量评测报告
X.3 回归执行目标

P3 回归执行只回答以下问题：

当前实现是否仍满足冻结状态边界
当前实现是否仍能正确解析并绑定引用
当前实现是否对 hard fail 保持稳定
当前版本相较上一版本是否出现边界漂移
失败是否集中在某一类样本或某一类路径

P3 回归执行 不回答：

模型整体效果提升了多少
用户体验是否更好
回答是否更自然
语义正确率是否更高

这些问题不属于 P3 回归执行的职责。

X.4 最小回归流程（冻结）

P3 benchmark 的最小执行流程固定为：

加载 benchmark manifest
    ↓
按样本列表逐条加载 case
    ↓
构造 ContextPool 与 llm_output
    ↓
执行 P3 网关
    ↓
得到实际 P3Result
    ↓
与 expected 做字段比对
    ↓
记录通过 / 失败
    ↓
汇总统计
    ↓
输出标准报告

该流程中，不允许插入：

外部检索补全
人工修正引用
动态修改样本 expected
模型参与解释

回归流程必须保持纯离线、纯确定性。

X.5 回归执行输入约束

每次回归执行，输入必须明确固定：

benchmark_version
case_set
runner_version
code_version（建议记录 git commit 或版本号）

其中：

benchmark_version 表示样本集版本
runner_version 表示执行器版本
code_version 表示被测实现版本

如果这些信息缺失，则同一次回归结果在后续复盘中将无法准确对齐。

X.6 单条样本执行规则

对每条 case，执行流程固定为：

读取 context_pool
读取 llm_output
构造 HostP3Request
调用 gateway.run()
获取 P3Result
与 expected 做字段比对
生成该 case 的 pass / fail 记录

单条样本执行必须完全独立。
前一条 case 的执行状态不得影响后一条 case。

因此，P3 benchmark runner 不允许：

共享跨 case 的缓存状态
复用上条 case 的 pool
依赖上一次执行结果做推断

每条 case 都必须视为一个独立的确定性判定单元。

X.7 结果比对分层（冻结）

为了避免回归过于脆弱，同时又保护核心边界，本章定义三层比对规则。

X.7.1 一级必比字段（硬比较）

以下字段必须严格一致：

reference_integrity_status

这是 P3 回归中最核心的字段。
只要该字段不一致，该样本即判定为 硬失败（hard fail）。

原因很简单：

状态边界一旦漂移，就说明 P3 的核心判定已经改变。

X.7.2 二级关键字段（关键集合比较）

以下字段建议做严格集合比较：

resolved_members
unresolved_members
outside_pool_references

这些字段用于判断：

状态虽然一样，但边界是否已经漂移
resolver 是否悄悄“多认”或“少认”了某些引用

二级字段不一致时，判定为 中失败（medium fail）。

也就是说：

主状态没变
但结构边界已经出现变化

这种情况不能忽略，但也不一定像状态漂移那样严重。

X.7.3 三级解释字段（辅助比较）

以下字段作为辅助比较项：

triggers
reasons
metrics

其中：

triggers 建议比较，但允许在受控变更下进行解释层升级
reasons 默认不作为阻塞项
metrics 默认不作为阻塞项

三级字段不一致时，判定为 软失败（soft fail） 或 提示性漂移（informational drift）。

其作用主要是帮助维护者发现解释层是否退化，而不是直接阻塞所有变更。

X.8 通过 / 失败判定规则（冻结）

本章将回归结果固定为四类：

X.8.1 Pass

以下条件全部满足：

一级字段一致
二级字段一致
三级字段无阻塞差异

即判定为通过。

X.8.2 Soft Fail

满足以下条件：

一级字段一致
二级字段一致
三级字段存在差异

例如：

triggers 少了一个解释码
reasons 文案发生变化
metrics 字段略有调整

Soft Fail 一般不应直接阻塞主线，但必须在报告中明确标出。

X.8.3 Medium Fail

满足以下条件：

一级字段一致
二级关键字段不一致

例如：

状态仍为 partial
但 resolved_members 结果发生变化
或 outside_pool_references 集合不一致

Medium Fail 表示主状态暂未漂移，但实际边界已发生变化，需要明确审查。

X.8.4 Hard Fail

满足以下任一条件：

reference_integrity_status 不一致
原本 hard fail 类样本被错误降级
当前实现违反设计冻结边界

例如：

outside_pool 被错误判成 partial
fabricated 被错误判成 unresolved
unresolved 被错误判成 clean

Hard Fail 应视为阻塞性失败。

X.9 回归执行策略建议

建议将 P3 回归执行分为两种模式。

X.9.1 快速回归模式

适用于：

小改 parser
小改 resolver
小改 evaluator
本地开发验证

特点：

跑全量 benchmark
只输出简要统计
失败只展示摘要
X.9.2 完整报告模式

适用于：

提交前检查
合并前检查
版本冻结前检查
规则升级验证

特点：

跑全量 benchmark
输出完整 category 统计
输出失败明细
输出 expected vs actual 对照
输出版本信息

当前阶段建议至少保证：

每次影响 P3 的修改，都要跑完整 benchmark。

X.10 标准报告结构（冻结）

建议 P3 benchmark 报告至少包含以下部分。

X.10.1 报告头
P3 Benchmark Report
benchmark_name: p3_reference_integrity
benchmark_version: v1
runner_version: v1
code_version: <git commit or release tag>
X.10.2 总体统计
Summary
- total_cases: 25
- pass: 21
- soft_fail: 2
- medium_fail: 1
- hard_fail: 1
X.10.3 分类统计
By Category
- clean: total=5, pass=5, fail=0
- partial: total=5, pass=4, fail=1
- unresolved: total=5, pass=4, fail=1
- outside_pool: total=5, pass=5, fail=0
- fabricated: total=5, pass=3, fail=2

分类统计的作用是避免只看总体通过率。
即使总体通过率很高，如果某一类失败集中，也应视为边界退化信号。

X.10.4 失败明细

每条失败样本至少输出：

case_id
category
failure_level
expected
actual
diff_summary

示例：

Failed Cases

1. case_id: p3_fabricated_001
   category: fabricated
   failure_level: hard_fail
   expected.status: fabricated
   actual.status: unresolved
   diff_summary: fabricated hard-fail path regressed to unresolved

2. case_id: p3_partial_003
   category: partial
   failure_level: medium_fail
   expected.resolved_members: ["doc_1"]
   actual.resolved_members: []
   diff_summary: resolver no longer recognizes valid member_id
X.10.5 结论区

报告结尾应给出简短结论，例如：

Conclusion
- hard_fail detected: yes
- release_blocking: yes
- major regression category: fabricated

或者：

Conclusion
- hard_fail detected: no
- release_blocking: no
- notes: only soft trigger drift observed

这样报告不仅能看明细，也能直接服务工程决策。

X.11 回归结果的阻塞规则（建议冻结）

建议将 P3 benchmark 的阻塞口径固定为：

阻塞条件

以下任一条件满足，即视为阻塞：

存在 hard_fail
同一 category 出现集中 medium_fail
hard fail 类样本（outside_pool / fabricated）出现状态漂移
非阻塞但需关注

以下情况一般不直接阻塞：

少量 soft_fail
reasons 文案变化
metrics 字段优化
需人工审查

以下情况建议触发人工审查：

多个 medium_fail
同一触发器相关样本连续漂移
某一类样本通过率明显下降
X.12 不允许的回归执行做法

这部分是本章红线。

X.12.1 禁止只看总通过率

错误做法：

“通过率 92%，所以没问题。”

P3 回归必须看：

失败集中在哪类
是否出现 hard fail
是否是 hard fail 路径退化

总通过率只能作为辅助信息，不能作为唯一结论。

X.12.2 禁止静默改 expected

当某条 case 跑不过时，不能直接改样本 expected 来“修平结果”。
必须先回答：

代码是否错了
样本真值是否错了
设计边界是否正式升级了

没有明确理由，不允许调整 expected。

X.12.3 禁止跳过 hard fail 类样本

为了“让报告更好看”，不能跳过：

outside_pool
fabricated

这两类正是 P3 的核心保护路径。
一旦跳过，P3 的 benchmark 将失去存在意义。

X.12.4 禁止把 soft fail 伪装成 pass

如果 triggers、解释字段、关键附加信息发生变化，不能直接吞掉。
即使不阻塞，也应如实记录为 soft fail 或 drift。

否则报告会丧失解释力。

X.12.5 禁止无版本执行

没有 benchmark_version / runner_version / code_version 的回归结果，不应进入正式归档。
否则后续无法做版本对照。

X.13 建议的最小报告输出形式

当前阶段不需要一开始就做复杂前端展示。
建议优先支持两种报告形式：

X.13.1 控制台摘要

适合本地快速回归：

总数
通过数
失败数
category 统计
简短失败清单
X.13.2 Markdown 报告

适合归档与评审：

结构清晰
可直接提交到仓库
可用于 PR / 设计评审记录

如果后续需要，再扩展到：

JSON 明细报告
HTML 报告
可视化 dashboard

但这些都不是当前阶段主线。

X.14 建议的最小 Markdown 报告模板
# P3 Benchmark Report

- benchmark_name: p3_reference_integrity
- benchmark_version: v1
- runner_version: v1
- code_version: <git commit>

## Summary
- total_cases: 25
- pass: 21
- soft_fail: 2
- medium_fail: 1
- hard_fail: 1

## By Category
- clean: total=5, pass=5, fail=0
- partial: total=5, pass=4, fail=1
- unresolved: total=5, pass=4, fail=1
- outside_pool: total=5, pass=5, fail=0
- fabricated: total=5, pass=3, fail=2

## Failed Cases

### p3_fabricated_001
- category: fabricated
- failure_level: hard_fail
- expected.status: fabricated
- actual.status: unresolved
- diff_summary: fabricated hard-fail path regressed to unresolved

## Conclusion
- hard_fail detected: yes
- release_blocking: yes
- notes: fabricated category needs review

这份模板的重点不是华丽，而是：

统一
易读
可归档
可比较
X.15 本章结论

P3 benchmark 如果只有样本而没有执行规范，就很容易退化为“有人手动跑过一次”的非工程化资产。
因此，本章冻结三条核心原则：

状态边界优先于表面通过率
失败分级优先于简单二元成败
报告标准化优先于临时口头结论

P3 的回归体系只有在“样本、执行、报告”三者同时标准化后，才能真正起到边界守卫作用。
