第 X 章：P3 Benchmark 与回归样本约束
X.1 文档目的

本章用于定义 P3 的 benchmark 与回归样本边界，目标不是追求“好看分数”，而是保证：

P3 的核心判定可重复验证
规则迭代不会破坏既有边界
新增解析能力不会偷偷改变主线定义
clean / partial / unresolved / outside_pool / fabricated 五类状态始终保持稳定

本章的核心思想是：

P3 的 benchmark 不是语义评测，而是引用归属评测。

因此，P3 的测试与回归样本不得被设计成“答案是否正确”的综合评测，而必须严格围绕：

引用是否可解析
引用是否可归属
引用是否越过当前 ContextPool
引用是否伪造
X.2 Benchmark 的定位

P3 benchmark 只回答以下问题：

当前解析器能否稳定提取引用
当前 resolver 能否稳定绑定到 ContextPool
当前 evaluator 能否稳定输出冻结状态枚举
规则修改后，历史样本是否仍保持相同判定
P3 是否仍然保持“纯结构、纯本地、纯确定性”的设计边界

P3 benchmark 不回答：

结论是否真实世界正确
文本是否被上下文语义支持
模型推理是否充分
用户是否会接受这个答案

这些都不是 P3 benchmark 的范围。

X.3 基本原则（冻结）
X.3.1 样本必须围绕 ContextPool 构建

每条 benchmark 样本都必须显式给出：

context_pool
llm_output
expected_status

必要时补充：

expected_triggers
expected_resolved_members
expected_unresolved_members
expected_outside_pool_references

不允许只给一句模型输出，然后让测试运行时自行猜测上下文。

X.3.2 样本真值必须人工可判定

每条样本必须做到：

不依赖主观解释，人工一眼就能判断对错

例如：

[ref:doc_1] 且 doc_1 在 ContextPool 中 → clean
[ref:missing_9] 且 missing_9 不在 ContextPool 中 → unresolved 或 partial
[ref:outside:secret_doc] → outside_pool
[ref:fake:made_up_doc] → fabricated

如果一条样本需要靠“语义理解”才能判断，那说明它不适合作为 P3 benchmark 样本。

X.3.3 样本必须可复现

同一条样本在任何时间、任何机器、任何环境下，只要输入相同：

ContextPool
llm_output
规则版本

就必须得到同样的 P3Result。

不允许：

依赖外部检索
依赖数据库状态
依赖网络资源
依赖模型温度或随机性
X.4 样本结构约束

建议将 benchmark 样本收敛为统一结构。

X.4.1 样本建议结构
{
    "case_id": "p3_clean_001",
    "description": "single valid reference inside pool",
    "context_pool": {
        "pool_id": "pool_1",
        "members": [
            {"member_id": "doc_1", "source_id": "s1", "content": "aaa"},
            {"member_id": "doc_2", "source_id": "s2", "content": "bbb"},
        ],
    },
    "llm_output": "结论来自 [ref:doc_1]",
    "expected": {
        "reference_integrity_status": "clean",
        "resolved_members": ["doc_1"],
        "unresolved_members": [],
        "outside_pool_references": [],
        "triggers": [],
    },
}
X.4.2 样本字段最小要求

每条样本至少必须包含：

case_id
context_pool
llm_output
expected.reference_integrity_status

推荐补充：

description
expected.triggers
expected.resolved_members
expected.unresolved_members
expected.outside_pool_references
X.4.3 case_id 必须稳定

case_id 必须：

全局唯一
长期稳定
不因描述优化而改变

因为回归系统最终依赖 case_id 跟踪历史结果变化。
禁止把 case_id 设计成可读描述本身。

建议命名规则：

p3_<status>_<serial>

例如：

p3_clean_001
p3_partial_003
p3_unresolved_002
p3_outside_pool_001
p3_fabricated_001
X.5 样本分类（冻结）

P3 benchmark 必须至少覆盖以下五大类。

X.5.1 Clean 类样本

目标：验证所有引用都能被正确解析并绑定到 ContextPool。

覆盖点：

单引用命中
多引用全部命中
引用顺序不同但结果一致
重复引用同一成员
无显式引用情况下的当前策略行为

该类样本用于防止：

正常合法引用被误判
解析器升级后误伤基础路径
X.5.2 Partial 类样本

目标：验证“部分能归属、部分不能归属”的混合情况。

覆盖点：

一个命中，一个缺失
多个命中，少量缺失
同时存在解析成功与无法解析
同时存在 pool 内合法引用与普通缺失引用

该类样本用于防止：

evaluator 把 partial 错误打成 clean
或把 partial 直接扩大成 outside_pool
X.5.3 Unresolved 类样本

目标：验证“引用无法被可靠解析”的情况。

覆盖点：

格式无法解析
引用 ID 缺失
引用字段不完整
全部引用都无法命中 ContextPool

该类样本用于防止：

parser 过度放宽
resolver 盲目补全
evaluator 把 unresolved 错判成 clean 或 partial
X.5.4 Outside Pool 类样本

目标：验证明确池外引用的硬失败路径。

覆盖点：

明确 outside 前缀
引用指向当前 pool 外部对象
解析成功但归属集合不属于本轮 ContextPool
多引用里混入一个池外对象

该类样本用于防止：

P3 被误降级成“只做提醒，不做硬约束”
主程序聚合时错误放过 hard fail
X.5.5 Fabricated 类样本

目标：验证伪造引用路径。

覆盖点：

明确 fake 前缀
结构上看似像引用，但明确不是合法池内对象
特意构造出来的伪成员
混合真实引用与伪造引用

该类样本用于防止：

fabricated 被误判为 unresolved
fabricated 丢失其硬失败语义
X.6 样本覆盖要求

P3 benchmark 不追求海量样本，但追求覆盖完整。
建议至少保证以下三层覆盖。

X.6.1 状态覆盖

五类状态必须全部有样本：

clean
partial
unresolved
outside_pool
fabricated

任何状态缺失，都意味着该状态没有被回归系统保护。

X.6.2 路径覆盖

每种状态应至少覆盖：

单引用路径
多引用路径
边界路径

例如 partial 至少要有：

1 个合法 + 1 个缺失
2 个合法 + 1 个缺失
1 个合法 + 1 个 parse_failed
X.6.3 错误覆盖

以下触发器建议至少各有一条命中样本：

reference_member_not_found
citation_outside_context_pool
unresolved_reference_pointer
fabricated_reference_detected
reference_parse_failed

如果某个 trigger 没有样本覆盖，后续改动时就无法知道它是否被意外破坏。

X.7 回归判定规则（冻结）

P3 回归比较时，至少比较以下字段：

一级必比字段
reference_integrity_status
二级建议比对字段
resolved_members
unresolved_members
outside_pool_references
triggers
三级辅助字段
metrics
reasons

其中：

status 是最核心字段，必须严格一致
resolved_members / unresolved_members / outside_pool_references 用于判断边界是否漂移
triggers 用于判断解释层是否退化
reasons 可允许文本优化，但不应改变核心含义
X.8 不允许的 benchmark 做法

这部分是本章重点。

X.8.1 禁止把 P3 benchmark 做成语义题

错误示例：

“回答是否被原文支持”
“这个结论是否合理”
“引用的句子是否与结论一致”

这些都属于 P2，不属于 P3。

X.8.2 禁止依赖外部环境

P3 benchmark 不允许依赖：

向量库
检索接口
数据库查询
HTTP 请求
模型推理结果

否则回归将不再可重复。

X.8.3 禁止把样本写成模糊题

错误示例：

“这条引用看起来可能对应 doc_2”
“语义上大概是这个文档”
“也许可以算作 partial”

P3 benchmark 必须是确定题，不允许“也许正确”。

X.8.4 禁止回归时动态改 expected

一旦某条样本的真值被确认，就不能因为代码跑不过而去改 expected。
只有在以下两种情况下才允许调整 expected：

先前真值定义错误
设计文档边界被正式升级，并经过显式版本变更

否则就会把 benchmark 变成“为当前代码洗地”。

X.8.5 禁止用 benchmark 掩盖设计漂移

如果一次 parser / resolver 修改导致大量样本状态变化，不能只看“通过率是否更高”。
必须先回答：

这次变化是否仍然符合 P3 的原始设计边界？

如果答案不明确，则不能接受该改动。

X.9 回归分层建议

建议把回归分成三层。

X.9.1 L1：单元样本回归

针对 parser / resolver / evaluator 的最小样本测试。
特点：

快
粒度细
易定位问题
X.9.2 L2：端到端 P3 回归

直接输入：

ContextPool
llm_output

输出：

P3Result

特点：

验证整条 P3 链路
是最重要的主回归层
X.9.3 L3：主程序聚合回归

把 P3 结果与 P2 结果一并送入主程序聚合器，检查：

是否正确 block / warn / allow
hard fail 是否被保住

这一层不是 P3 自身逻辑测试，但用于验证 P3 没有在主程序集成时失效。

X.10 样本版本管理约束

建议对 benchmark 样本采用“样本版本”概念。

X.10.1 样本版本号

例如：

p3_benchmark_version = v1

用于标识当前样本集的冻结状态。

X.10.2 变更原则

新增样本可以是正常演进；
修改已有样本真值必须满足：

有明确变更说明
有边界变化理由
有版本记录
X.10.3 不允许静默替换

不允许不记录说明，直接替换样本内容或 expected 结果。
否则历史对比将失去意义。

X.11 建议的最小 benchmark 规模

在当前阶段，P3 不需要一开始就做大量样本。
建议最小可用集为：

clean：5 条
partial：5 条
unresolved：5 条
outside_pool：5 条
fabricated：5 条

即：

最小 25 条高确定性样本

这 25 条的价值远高于 200 条边界含糊的样本。

X.12 本章结论

P3 benchmark 的本质不是“评估回答质量”，而是：

保护 P3 的结构边界不被规则升级、解析增强或实现漂移悄悄破坏。

因此，本章冻结三条底线：

样本必须围绕 ContextPool 构建。
真值必须人工可判定。
回归必须优先保护状态边界，而不是追求表面通过率。

只要这三条不变，P3 就能在后续迭代中保持清晰、稳定、可审计。
