# P3 回归执行规范与结果报告模板

## 7.1 本章目标

本章用于定义 P3 benchmark 的执行方式、结果判定方式与报告输出方式，目标不是追求“跑完就算”，而是保证：

- benchmark 可重复执行；
- 回归结果可审计；
- 失败原因可定位；
- 历史版本可比较；
- 规则迭代不会因为报告口径不一致而失真。

本章的核心目标，是把 P3 的 benchmark 从“样本集合”推进为“标准化回归流程”。

---

## 7.2 适用范围

本章适用于：

- P3 benchmark 的日常执行；
- parser / resolver / evaluator 修改后的回归验证；
- 冻结版本之间的结果比较；
- benchmark 报告归档与复盘。

本章不适用于：

- P2 语义评测执行规范；
- 主程序完整集成压测；
- 行业专项插件评测流程；
- 模型质量评测报告。

---

## 7.3 回归执行目标（冻结）

P3 回归执行只回答以下问题：

1. 当前实现是否仍满足冻结状态边界；
2. 当前实现是否仍能正确解析并绑定引用；
3. 当前实现是否对 hard fail 保持稳定；
4. 当前版本相较上一版本是否出现边界漂移；
5. 失败是否集中在某一类样本或某一类路径。

P3 回归执行不回答：

- 模型整体效果提升了多少；
- 用户体验是否更好；
- 回答是否更自然；
- 语义正确率是否更高；
- 最终业务价值是否提升。

这些问题不属于 P3 回归执行的职责。

---

## 7.4 最小回归流程（冻结）

P3 benchmark 的最小执行流程固定为：

```text
加载 benchmark manifest
    ↓
按样本列表逐条加载 case
    ↓
构造 ContextPool 与 llm_output
    ↓
执行 P3 网关
    ↓
得到实际 P3Result
    ↓
与 expected 做字段比对
    ↓
记录通过 / 失败
    ↓
汇总统计
    ↓
输出标准报告

该流程中，不允许插入：

外部检索补全；
人工修正引用；
动态修改样本 expected；
模型参与解释；
任意基于当前结果的临时补救逻辑。

回归流程必须保持：

纯离线；
纯确定性；
可重复；
可归档。
7.5 回归执行输入约束（冻结）

每次回归执行，输入必须明确固定：

benchmark_version
case_set
runner_version
code_version

其中：

7.5.1 benchmark_version

表示样本集版本。
用于标识当前运行所使用的 benchmark 真值集。

7.5.2 case_set

表示本次实际执行的样本集合。
允许是全量集合，也允许是明确说明的子集，但必须被记录。

7.5.3 runner_version

表示执行器版本。
用于区分“样本真值没变，但 runner 统计口径变了”的情况。

7.5.4 code_version

表示被测实现版本。
建议记录 git commit、tag 或明确版本号。

如果这些信息缺失，则回归结果在后续复盘中将无法可靠对齐。

7.6 单条样本执行规则（冻结）

对每条 case，执行流程固定为：

读取 context_pool
读取 llm_output
构造 HostP3Request
调用 gateway.run()
获取 P3Result
与 expected 做字段比对
生成该 case 的 pass / fail 记录

单条样本执行必须完全独立。

前一条 case 的执行状态不得影响后一条 case。
因此，P3 benchmark runner 不允许：

共享跨 case 的缓存状态；
复用上条 case 的 pool；
依赖上一条执行结果做推断；
因上一条失败而修改后一条输入。

每条 case 都必须视为一个独立的确定性判定单元。

7.7 结果比对分层（冻结）

为了避免回归过于脆弱，同时又保护核心边界，本章定义三层比对规则。

7.7.1 一级必比字段（硬比较）

以下字段必须严格一致：

reference_integrity_status

这是 P3 回归中最核心的字段。
一旦该字段发生变化，就不再是“解释文字差异”，而是状态边界变化。

7.7.2 二级关键字段（强比较）

以下字段建议纳入强比较：

resolved_members
unresolved_members
outside_pool_references
triggers

这些字段用于帮助判断：

状态为什么变化；
漂移发生在 parser、resolver 还是 evaluator；
问题是单条 case 误差，还是一类规则整体漂移。
7.7.3 三级辅助字段（弱比较）

以下字段可作为弱比较或附加比较：

reasons
metrics
报告附加统计信息

这些字段如果变化，不一定代表边界失真；
但如果出现显著漂移，仍应在报告中记录，而不是直接吞掉。

7.8 失败分级（冻结）

为了避免回归结果退化为“只看 pass / fail”的粗糙口径，本章固定失败分级如下。

7.8.1 pass

核心状态与关键字段均符合预期。
可视为稳定通过。

7.8.2 soft_fail

一级核心字段未变，但二级或三级字段出现差异。
例如：

reference_integrity_status 正确；
但 triggers 与 expected 不一致；
或解释字段发生偏移。

soft_fail 不一定阻塞发布，但必须记录。

7.8.3 medium_fail

一级核心字段正确，但二级关键字段出现较大差异，说明实现可能存在局部退化。
例如：

status = partial 保持不变；
但 resolved_members / unresolved_members 显著偏移；
或 category 内多条 case 出现一致性异常。

medium_fail 通常需要人工复查，不应被当作普通噪声忽略。

7.8.4 hard_fail

一级核心字段发生错误，尤其是以下情况：

clean 被打成 unresolved
outside_pool 被降级为 partial
fabricated 被降级为 unresolved
hard fail 路径失去阻断语义

hard_fail 应视为边界回归失败，默认进入发布阻塞或强审查路径。

7.9 建议的失败判定规则

建议采用如下最小判定方式：
```python
def judge_case_result(expected, actual):
    if actual.reference_integrity_status != expected.reference_integrity_status:
        return "hard_fail"

    key_fields_changed = any([
        actual.resolved_members != expected.get("resolved_members", actual.resolved_members),
        actual.unresolved_members != expected.get("unresolved_members", actual.unresolved_members),
        actual.outside_pool_references != expected.get("outside_pool_references", actual.outside_pool_references),
    ])

    trigger_changed = actual.triggers != expected.get("triggers", actual.triggers)

    if key_fields_changed:
        return "medium_fail"

    if trigger_changed:
        return "soft_fail"

    return "pass"
```
