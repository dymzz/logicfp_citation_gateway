# P3 版本演进与变更控制约束

## 8.1 本章目标

本章用于约束 P3 在后续迭代中的版本演进方式与变更控制方式，防止以下问题出现：

- parser 一次“小优化”偷偷改变状态边界；
- resolver 为了“更智能”突破 ContextPool 边界；
- benchmark 样本被静默修改，导致历史结果失真；
- 主程序接入方在没有显式版本升级的情况下被迫适配新口径；
- P3 从“引用归属网关”逐步漂移成“半语义判断器”。

本章的目标不是阻止演进，而是确保：

**P3 的任何演进都必须是显式的、可追踪的、可回归验证的。**

---

## 8.2 版本控制的核心原则（冻结）

### 8.2.1 边界稳定优先于功能扩张

P3 的首要目标不是“支持更多花样引用”，而是保持其核心定义长期稳定：

**P3 只判断引用是否属于当前 ContextPool。**

任何功能扩张，只要触碰这个定义，都必须视为高风险改动。

因此，P3 的演进顺序必须是：

1. 先保护边界；
2. 再扩展能力；
3. 最后优化实现。

而不能反过来。

### 8.2.2 公开 contract 稳定优先于内部实现自由

P3 允许内部实现调整，例如：

- parser 重构；
- resolver 优化；
- evaluator 代码整理；
- runner 性能改进。

但这些调整不得破坏已冻结的公开 contract，包括：

- `HostP3Request`
- `P3Result`
- `reference_integrity_status` 枚举
- trigger code 公开集合
- 主程序接入顺序与聚合边界

内部可以重构，公开 contract 不可随意漂移。:contentReference[oaicite:3]{index=3}

### 8.2.3 显式变更优先于隐式漂移

P3 不允许“边改边用，改了再说”的演进方式。  
任何影响以下内容的改动，都必须被视为显式变更：

- 状态边界
- 输出字段含义
- trigger 语义
- benchmark 真值
- 主程序集成口径

换句话说：

**没有明确变更说明的行为，不得改变 P3 的对外语义。** :contentReference[oaicite:4]{index=4}

## 8.3 版本分层（建议冻结）

建议把 P3 的版本拆成三层，分别管理。

### 8.3.1 设计版本（design_version）

用于标识 P3 设计文档和冻结边界的版本。

适用范围：

- 接口冻结变化
- 状态枚举变化
- trigger / error code 变化
- P2 / P3 / 主程序边界变化
- 核心禁止项变化

建议形式：

- `v1`
- `v1.1`
- `v2`

### 8.3.2 实现版本（implementation_version）

用于标识代码实现版本。

适用范围：

- parser / resolver / evaluator 实现变更
- runner 逻辑优化
- 性能优化
- 内部重构
- 非破坏性 bugfix

建议形式：

- `0.1.0`
- `0.2.0`
- `0.2.1`

### 8.3.3 benchmark 版本（benchmark_version）

用于标识 benchmark 样本集版本。

适用范围：

- 新增正式 case
- 删除正式 case
- 修改已有 case 的 expected
- manifest 结构变化
- 分类组织变化

建议形式：

- `v1`
- `v2`
- `v3`

这样分层的目的，是避免把“文档边界变化、代码实现变化、样本真值变化”混成一个模糊版本号。:contentReference[oaicite:5]{index=5}

## 8.4 变更分类（建议冻结）

建议将 P3 变更分为四类。

### 8.4.1 A 类：纯实现重构

特点：

- 不改变公开 contract；
- 不改变状态边界；
- 不改变 trigger / error code 语义；
- 不改变 benchmark 真值。

示例：

- parser 内部代码整理；
- evaluator 结构重构；
- runner 性能优化；
- 非对外语义的 bugfix。

要求：

- 升级 implementation_version；
- 执行 benchmark；
- 产出对比报告。

### 8.4.2 B 类：实现增强但不改变边界

特点：

- 功能更稳、更全、更易维护；
- 但对外语义不变；
- benchmark 核心状态不应漂移。

示例：

- parser 支持更多等价合法格式，但最终仍统一归约到同一 `ReferenceClaim`；
- metrics 扩展；
- 日志增强；
- 报告增强。

要求：

- 升级 implementation_version；
- 如 benchmark 集变化，同时升级 benchmark_version；
- 必须证明核心状态边界未漂移。

### 8.4.3 C 类：样本或执行规则变更

特点：

- 设计边界不一定变化；
- 但 benchmark 真值、样本覆盖、runner 规则或报告口径发生变化。

示例：

- 新增一组 fabricated 边界样本；
- 调整某类样本 expected；
- manifest 改版；
- 回归失败分级调整。

要求：

- 升级 benchmark_version；
- 如 runner 规则变化，升级 runner_version；
- 输出前后版本差异报告；
- 明确说明是否影响历史对比。

### 8.4.4 D 类：设计边界变化

特点：

- 触碰冻结边界；
- 触碰公开 contract；
- 触碰状态枚举；
- 触碰 P3 与主程序、P2 的职责划分。

示例：

- 修改 `reference_integrity_status` 集合；
- 修改 `HostP3Request` / `P3Result` 字段语义；
- 修改 outside_pool / fabricated / unresolved 的边界定义；
- 修改核心禁止项；
- 修改 P3 与主程序聚合的硬失败语义。

要求：

- 升级 design_version；
- 必须同步审查 benchmark_version 与 implementation_version；
- 必须给出迁移说明；
- 必须执行完整 benchmark 与对比报告；
- 必须人工审查。:contentReference[oaicite:6]{index=6}

## 8.5 必须升级版本号的场景

### 8.5.1 必须升级 implementation_version 的情况

包括但不限于：

- parser / resolver / evaluator 代码逻辑变化；
- P3 runner 逻辑变化；
- 输出生成逻辑变化；
- 内部 bugfix；
- 性能优化。

只要代码实现变了，就应升级 implementation_version。

### 8.5.2 必须升级 benchmark_version 的情况

包括：

- 正式新增 benchmark case；
- 删除已有正式 case；
- 调整某条 case 的 expected；
- 调整 manifest 结构；
- 调整 benchmark 分类组织规则。

只要样本集本身变了，就应升级 benchmark_version。:contentReference[oaicite:7]{index=7}

### 8.5.3 必须升级 design_version 的情况

包括：

- 修改冻结 contract；
- 修改状态枚举；
- 修改 P3 与主程序的职责边界；
- 修改 P3 与 P2 的边界；
- 修改核心禁止项；
- 修改状态判定语义。

只要设计口径变了，就不能只改代码或样本，必须升级 design_version。:contentReference[oaicite:8]{index=8}

## 8.6 变更提交流程（建议冻结）

建议所有影响 P3 的正式变更都遵循以下流程：

```text
提出变更意图
    ↓
判定变更类型（A / B / C / D）
    ↓
更新版本号
    ↓
修改实现或文档
    ↓
执行完整 benchmark
    ↓
输出对比报告
    ↓
人工审查是否接受

这个流程的重点不在形式，而在于：

每次改动都必须被分类；
每次改动都必须被验证；
每次改动都必须留下可追踪证据。
8.7 变更说明模板（建议冻结）

每次正式变更，建议至少附带以下字段：
change_id:
change_type: A / B / C / D
design_version:
implementation_version:
benchmark_version:
summary:
affected_scope:
expected_impact:
benchmark_result:
breaking_change: yes / no
migration_required: yes / no

8.7.1 示例
change_id: p3-change-2026-001
change_type: C
design_version: v1
implementation_version: 0.4.0
benchmark_version: v2
summary: refine fabricated detection rule for explicit fake-prefixed references
affected_scope: evaluator, fabricated benchmark cases
expected_impact: fabricated cases may change trigger details, status should remain stable
benchmark_result: no hard fail, 2 soft fail
breaking_change: no
migration_required: no

这次改了什么；
影响了哪一层；
是否触碰冻结边界；
是否需要迁移

8.8 Benchmark 与变更控制的关系

P3 的 benchmark 不只是测试集，更是变更控制工具。
任何 P3 变更，都必须借助 benchmark 回答两个问题：

这次改动是否改变了核心边界；
这次改动是否破坏了历史冻结样本。

因此，本章明确规定：

没有 benchmark 对比报告的 P3 变更，不应视为完整变更。

尤其是 C 类和 D 类变更，不能只靠“逻辑上应该没问题”来放过。

8.9 回归分层建议

建议把回归分成三层。

8.9.1 L1：单元样本回归

针对 parser / resolver / evaluator 的最小样本测试。

特点：

快；
粒度细；
易定位问题。
8.9.2 L2：端到端 P3 回归

直接输入：

ContextPool
llm_output

输出：

P3Result

特点：

验证整条 P3 链路；
是最重要的主回归层。
8.9.3 L3：主程序聚合回归

把 P3 结果与 P2 结果一并送入主程序聚合器，检查：

是否正确 block / warn / allow
hard fail 是否被保住

这一层不是 P3 自身逻辑测试，但用于验证 P3 没有在主程序集成时失效。

8.10 样本版本管理约束

建议对 benchmark 样本采用“样本版本”概念。

8.10.1 样本版本号

例如：

p3_benchmark_version = v1

用于标识当前样本集的冻结状态。

8.10.2 变更原则

新增样本可以是正常演进；
修改已有样本真值必须满足：

有明确变更说明；
有边界变化理由；
有版本记录。
8.10.3 不允许静默替换

不允许不记录说明，直接替换样本内容或 expected 结果。
否则历史对比将失去意义。

8.11 建议的最小 benchmark 规模

在当前阶段，P3 不需要一开始就做大量样本。
建议最小可用集为：

clean：5 条
partial：5 条
unresolved：5 条
outside_pool：5 条
fabricated：5 条

即：

最小 25 条高确定性样本。

这 25 条的价值，远高于 200 条边界含糊的样本。

8.12 不允许的变更控制做法（冻结）

这部分是本章红线。

8.12.1 禁止静默修改 benchmark 真值

当代码跑不过时，不能先改 expected 再说。
必须先回答：

当前代码是否违背原冻结边界；
样本真值是否写错；
是否发生了正式设计升级。

如果没有正式变更依据，不得改 benchmark 真值。

8.12.2 禁止用实现现状反推设计边界

不能因为当前实现“已经这么跑了”，就倒推出：

状态边界应该改；
trigger 定义应该改；
hard fail 应该放松。

设计边界必须由设计文档和附录显式控制，而不是由实现现状倒逼。

8.12.3 禁止无版本记录地改公开 contract

不得在没有 design_version 变更记录的情况下，直接修改：

HostP3Request
P3Result
状态枚举
trigger 公开集合
聚合边界

否则主程序集成方会在无预警下被迫适配。

8.12.4 禁止把 P3 演进成半语义判断器

只要改动开始依赖：

embedding
LLM
外部检索
模糊语义补全

来判断引用归属，就已经触碰 P3 的设计红线。
这类改动不应作为“普通增强”进入 P3。
它要么被拒绝，要么必须被重新归类为别的能力层。

8.12.5 禁止没有 benchmark 对比报告就接受高风险变更

尤其是以下变更：

状态边界变化；
hard fail 路径变化；
fabricated / outside_pool / unresolved 分界变化；
benchmark 真值变化；
主程序集成语义变化。

没有 benchmark 对比报告，不应视为完整变更。

8.13 变更审查的最小问题清单

每次准备接受正式变更时，建议至少回答以下问题：

这次变化是否仍符合 P3 的原始设计边界；
是否触碰了 ContextPool 作为唯一合法边界来源这一原则；
是否影响公开 contract；
是否影响 benchmark 真值；
是否影响主程序集成口径；
是否需要 design_version 升级；
是否已经产出 benchmark 对比报告；
是否需要迁移说明。

如果这些问题中有关键项回答不清，则不应接受该变更。

8.14 与附录、样本、回归执行的关系

本章只负责：

规定 P3 如何演进；
规定变更如何分类；
规定什么情况下必须升级版本；
规定 benchmark 如何参与变更控制。

本章不重新定义：

状态枚举；
trigger / error code；
单条样本模板；
回归执行细节；
主程序最终业务决策。

8.15 本章结论

P3 的版本演进如果没有明确控制，就很容易在“看起来只是小优化”的过程中，慢慢失去最初的边界定义。

因此，本章冻结四条底线：

边界稳定优先于功能扩张；
公开 contract 稳定优先于内部实现自由；
显式变更优先于隐式漂移；
没有 benchmark 对比报告的高风险变更，不应视为完整变更。

只要这四条守住，P3 就能在后续迭代中保持：

清晰；
稳定；
可追踪；
可回归；
可审计。
