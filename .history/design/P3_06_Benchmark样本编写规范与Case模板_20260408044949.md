第 X 章：P3 Benchmark 样本编写规范与 Case 模板
X.1 文档目的

本章用于统一 P3 benchmark 样本的编写方式，解决三个问题：

什么样的 case 才算合格的 P3 样本
case 应该按什么格式落盘
编写新样本时必须遵守哪些约束

本章不是定义 P3 的判定规则，而是定义：

如何把已经冻结的判定规则，稳定地写成可回归、可维护、可扩展的 benchmark 样本。

X.2 适用范围

本章仅适用于 P3：

引用解析
引用归属
池内 / 池外判定
fabricated / unresolved / partial / clean 等状态回归

本章不适用于：

P2 语义支持评测
主程序最终 verdict 评测
行业专项规则样本
LLM 主观质量评测
X.3 样本编写总原则（冻结）
X.3.1 一条 case 只验证一个核心边界

每条 case 应尽量聚焦一个主要目标。
不要在同一条 case 里同时塞入太多变量，否则失败后无法快速定位。

例如：

好 case：验证单个合法引用是否能命中 pool
坏 case：同时混入合法引用、池外引用、格式错误、fabricated、业务规则

允许有混合 case，但必须明确“主验证目标”是什么。

X.3.2 case 必须人工可读、可判

任何一个 case，工程同事在不运行代码的情况下，也应能快速看出：

当前 pool 里有什么
输出里引用了什么
预期为什么是这个状态

如果一条 case 需要长篇解释才能理解，说明 case 设计过于复杂。

X.3.3 case 必须最小化

每条样本只保留判定所需的最小信息：

最小 pool
最小 llm_output
最小 expected

不要加入与当前判定无关的长文本、冗余 metadata、复杂描述。
否则会增加维护成本，降低可读性。

X.3.4 expected 必须先于实现存在

编写样本时，应先写清楚 expected，再跑代码。
不允许先看当前实现输出，再反过来写 expected。

否则 benchmark 会从“验证设计”退化为“迎合现状”。

X.4 样本文件组织规范

建议 P3 benchmark 样本采用独立目录，按状态分类存放。

推荐结构：

benchmarks/
└── p3/
    ├── clean/
    │   ├── p3_clean_001.json
    │   ├── p3_clean_002.json
    │   └── ...
    ├── partial/
    │   ├── p3_partial_001.json
    │   └── ...
    ├── unresolved/
    │   ├── p3_unresolved_001.json
    │   └── ...
    ├── outside_pool/
    │   ├── p3_outside_pool_001.json
    │   └── ...
    ├── fabricated/
    │   ├── p3_fabricated_001.json
    │   └── ...
    └── manifest.json

这样做的好处：

一眼可见状态覆盖是否完整
新样本落位明确
回归结果按类别统计更清楚
X.5 单条 Case 的标准结构（冻结）

建议每条样本采用统一 JSON 结构。

X.5.1 标准字段
{
  "case_id": "p3_clean_001",
  "description": "single valid reference inside context pool",
  "category": "clean",
  "context_pool": {
    "pool_id": "pool_1",
    "members": [
      {
        "member_id": "doc_1",
        "source_id": "source_a",
        "title": "Doc 1",
        "content": "This is document 1.",
        "metadata": {}
      }
    ],
    "retrieval_meta": {}
  },
  "llm_output": "结论来自 [ref:doc_1]",
  "expected": {
    "reference_integrity_status": "clean",
    "resolved_members": ["doc_1"],
    "unresolved_members": [],
    "outside_pool_references": [],
    "triggers": []
  },
  "notes": "baseline valid case"
}
X.6 字段规范（冻结）
X.6.1 case_id

要求：

全局唯一
长期稳定
不得复用
不得因文字润色而修改

推荐格式：

p3_<category>_<serial>

示例：

p3_clean_001
p3_partial_004
p3_unresolved_002

禁止：

test1
new_case
maybe_ok_case
含糊、临时、不可追踪命名
X.6.2 description

要求：

简短
客观
描述 case 目标，不写推理散文

好例子：

single valid reference inside context pool
one valid reference and one missing reference
explicit outside-pool reference

坏例子：

this case is a little tricky and should maybe be partial
I think this should probably fail
X.6.3 category

只能取以下五类之一：

clean
partial
unresolved
outside_pool
fabricated

该字段用于：

样本落盘目录校验
覆盖率统计
样本组织
回归报告分组

category 必须与 expected.reference_integrity_status 保持一致。

X.6.4 context_pool

必须是完整、显式、可自解释的数据。
禁止省略 pool，然后依赖外部 fixture 自动拼接。

必需字段
pool_id
members
推荐字段
retrieval_meta
members 中每个成员最少包含
member_id
source_id
content

可选字段：

title
metadata
X.6.5 llm_output

要求：

就是待测输出文本本身
不要混入解释说明
不要混入测试注释
必须能直接送入 parser

好例子：

结论来自 [ref:doc_1]

坏例子：

这里假设模型引用了 doc_1，所以应该是 clean
X.6.6 expected

expected 是样本真值定义区，必须简洁、稳定、结构化。

最少必须包含：

reference_integrity_status

推荐包含：

resolved_members
unresolved_members
outside_pool_references
triggers

不建议一开始强制比对：

reasons
metrics

因为这些字段可能随着实现优化而细化，但不应影响主判定。

X.6.7 notes

notes 是可选字段，只用于帮助维护者理解该样本的目的。
不能把 notes 当作 expected 的补充真值来源。

也就是说：

真值看 expected
补充说明看 notes

不能反过来。

X.7 Case 编写步骤（建议流程）

建议写 case 时遵循以下顺序：

第一步：先选分类

先明确这条样本属于哪一类：

clean
partial
unresolved
outside_pool
fabricated
第二步：最小化 ContextPool

只保留当前 case 判定所需成员。
不要无意义堆 10 个 members。

例如只测 doc_1，那 pool 里通常 1 到 2 个 member 就够了。

第三步：写最小 llm_output

只保留当前 case 需要的引用文本。
不要为了“更像真实回答”而写过长内容。

第四步：先写 expected

在脑中先判定这条 case 的状态，再把 expected 写出来。

第五步：最后补 description / notes

这一步只是提高可维护性，不影响真值。

X.9 推荐的 manifest 结构

建议在 benchmarks/p3/manifest.json 中维护样本清单，便于统计和校验。

示例：

{
  "benchmark_name": "p3_reference_integrity",
  "benchmark_version": "v1",
  "categories": {
    "clean": 5,
    "partial": 5,
    "unresolved": 5,
    "outside_pool": 5,
    "fabricated": 5
  },
  "total_cases": 25
}

这个文件的作用：

做覆盖率检查
做版本记录
防止某类样本不小心被删空
X.10 Case 编写禁止项（冻结）
X.10.1 禁止把多重目标塞进一条 case

不要在一条 case 中同时验证：

parse_failed
outside_pool
fabricated
partial

除非该样本明确就是“混合边界样本”，且有特殊目的。

X.10.2 禁止使用模糊真值

错误写法：

{
  "expected": {
    "reference_integrity_status": "maybe_partial"
  }
}

真值必须是冻结枚举之一，不能出现模糊词。

X.10.3 禁止依赖实现细节

case 不应依赖内部排序、内部数据结构、临时调试字段。
样本只依赖公开 contract。

X.10.4 禁止在 case 中夹带业务规则

P3 样本里不要出现：

法律条文优先级
医疗术语限制
审批流要求
组织策略

这些都不是 P3 的样本范围。

X.10.5 禁止为当前代码“洗样本”

如果某条样本与当前代码冲突，应先审查：

是代码错了
还是样本真值错了
还是设计边界变了

不能直接把样本改成“当前实现喜欢的样子”。

X.11 样本评审建议

新增 case 时，建议至少检查以下 6 项：

case_id 是否唯一
category 与 expected.status 是否一致
context_pool 是否足够小且完整
llm_output 是否能直接送入 P3
expected 是否人工可判
该 case 是否真的新增了覆盖，而不是重复已有 case

只有通过这 6 项，样本才应进入正式 benchmark 集。

X.12 本章结论

P3 benchmark 样本的质量，不取决于数量，而取决于是否满足三点：

结构统一
真值清晰
边界稳定

因此，本章冻结的核心要求是：

每条 case 都必须是一个最小、清晰、可人工判定的引用归属问题。

只要坚持这个原则，后续无论 parser、resolver 还是 evaluator 怎么演进，benchmark 都能持续充当 P3 的边界守卫。
