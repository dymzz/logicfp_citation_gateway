边界约束
接入时序
benchmark 约束
case 模板

现在就差把这些样本 怎么跑、怎么判定通过、怎么出报告 定下来。
这样 P3 设计文档就从“定义能力”变成“定义可验证工程流程”。

这章建议包含 4 部分：

1. 回归执行规范

定死最小流程：

加载 benchmark 样本
→ 逐条运行 P3
→ 生成实际输出
→ 与 expected 比对
→ 统计通过率
→ 输出分类报告

这里要写清：

一级必比字段：reference_integrity_status
二级比对字段：resolved_members / unresolved_members / outside_pool_references / triggers
reasons / metrics 默认不作为阻塞失败项
2. 通过 / 失败判定规则

建议定成三层：

硬失败：status 不一致
中失败：状态一致，但关键集合字段不一致
软失败：triggers 或说明字段轻微漂移

这样后面 parser 或 evaluator 微调时，不会因为文案变化把整套回归搞坏。

3. 回归报告模板

把输出格式写死，至少包括：

benchmark 版本
总样本数
各 category 数量
通过数 / 失败数
按 category 分组失败清单
每条失败样本的 expected vs actual

报告最小结构可以是：

P3 Benchmark Report
- version: v1
- total: 25
- passed: 23
- failed: 2

Failed Cases:
1. p3_partial_003
   expected.status = partial
   actual.status   = clean
2. p3_fabricated_001
   expected.trigger = fabricated_reference_detected
   actual.trigger   = unresolved_reference_pointer
4. 禁止项

这章一定要补这几条：

❌ 不允许只看总通过率，不看失败分布
❌ 不允许因为实现变化就静默改 expected
❌ 不允许把 hard fail 当作可接受波动
❌ 不允许跳过某些 category 只跑“容易通过”的样本

写完这一章后，P3 文档基本就完整了。
