《P3 最小接入时序图 + 宿主调用约束》

这一步的作用，不是再发明新规则，而是把前面已经冻结的东西，变成主程序怎么接 P3 的标准流程。
也就是把这条链彻底写死：

Host 构建 ContextPool
    ↓
Host 生成 HostP3Request
    ↓
P3 parse
    ↓
P3 resolve
    ↓
P3 evaluate
    ↓
返回 P3Result
    ↓
主程序与 P2 结果聚合
    ↓
输出统一 DecisionResult

你这一步文档里要补 4 个东西：

1. 宿主调用时序

明确谁先做、谁后做：

先构建 ContextPool
再拿 llm_output
再调用 P3
再把 P3 和 P2 聚合
最后才生成主程序 decision

这里最关键的是一句话要写死：

P3 必须在主程序最终放行前执行，不能后置为审计补充项。

2. 宿主侧前置条件

把“什么时候不允许调 P3”写出来：

必须满足
ContextPool 已构建完成
member_id 唯一
llm_output 是最终待放行文本
不允许对 llm_output 先做修复再送 P3
不满足时

不进入正常 P3 判断，直接进入主程序错误分支，例如：

ERR_CONTEXT_POOL_INVALID
ERR_CONTEXT_POOL_EMPTY
ERR_P3_INPUT_INVALID
3. 宿主调用禁止项

这是这一章最重要的部分，要明确写：

❌ 宿主不能跳过 P3 直接按 P2 放行
❌ 宿主不能在 P3 前先改写引用
❌ 宿主不能把检索系统当成 P3 的补全器
❌ 宿主不能把 P3 结果当“仅日志参考”
❌ 宿主不能在 outside_pool / fabricated 时继续 allow
4. 最小宿主接口

把主程序真正要依赖的调用方式固定住，例如：

p3_result = reference_gateway.run(p3_request)
decision = merge_p2_p3_to_decision(
    p2_result=p2_result,
    p3_result=p3_result,
)

这样后面无论你内部怎么重构，宿主只认这两个动作：

调 P3
合并决策
这一步完成后，P3 文档就基本闭环了

顺序应该是：

P3 定位与边界
P3 输入输出 contract
P3 禁止项
P2/P3 聚合规则
P3 最小接入时序与宿主调用约束

到这里，设计层就够了。
再往后才是可选项：

decision contract 单独成章
audit event 对接
benchmark 样本规范

本章不再讨论 P3 的能力边界，而是明确：

宿主系统在什么位置接入 P3
P3 的前置条件是什么
宿主必须遵守哪些调用约束
哪些做法属于错误接入
P3 返回后如何进入主程序统一决策链

本章目标是把 P3 从“独立能力模块”收敛为“主程序放行链路中的固定节点”。

X.2 接入定位

P3 不是离线审计器，也不是事后分析器。
P3 的定位是：

主程序最终放行前的引用完整性校验节点

也就是说，P3 必须位于“待放行回答”与“统一 decision 生成”之间。

P3 的职责不是生成最终业务响应，而是为主程序提供一个确定性的、可审计的引用完整性判断结果。

X.3 最小时序（冻结）

宿主侧最小时序固定为：

Host 构建 ContextPool
    ↓
Host 获取待放行 llm_output
    ↓
Host 生成 HostP3Request
    ↓
P3 Parser 解析引用
    ↓
P3 Resolver 绑定 ContextPool
    ↓
P3 Evaluator 输出 P3Result
    ↓
主程序聚合 P2Result + P3Result
    ↓
生成统一 DecisionResult
    ↓
主程序决定 allow / warn / fallback / block

该顺序中，P3 必须在统一 decision 生成之前执行。
不得把 P3 后置为“补充日志模块”或“事后告警模块”。

X.4 宿主前置条件

宿主在调用 P3 前，必须满足以下条件。

X.4.1 ContextPool 已构建完成

P3 的唯一判断边界来自 ContextPool。
如果 ContextPool 尚未完成构建，则 P3 无法定义“池内 / 池外”的边界。

因此，宿主必须保证：

当前轮可用上下文已经收口
当前轮允许引用的成员集合已经确定
ContextPool.members 为本轮放行的唯一合法引用集合
X.4.2 member_id 唯一且稳定

宿主必须保证每个 ContextMember.member_id：

非空
唯一
在本轮调用中稳定不变

若存在重复 member_id，则引用归属关系失去确定性。
这种情况不应交给 P3 处理，而应视为宿主输入错误。

X.4.3 llm_output 必须是最终待放行文本

送入 P3 的 llm_output 必须是：

主程序准备交付给用户或下游系统的最终文本

不允许把中间态文本、草稿文本、未经收口的增量文本送入 P3。
否则会导致：

P3 校验对象与最终输出对象不一致
审计链断裂
decision 结果不可追溯
X.4.4 宿主不得在 P3 前修改引用

宿主在调用 P3 前，不得对引用做自动修复、替换、重写、补全。
P3 必须面对模型原始待放行输出，而不是“被预处理过的输出”。

否则，P3 的判断对象将不再是模型真实产物。

X.5 宿主输入错误处理约束

当宿主前置条件不满足时，不进入正常 P3 判断流程。
这些情况应由主程序直接进入输入错误分支。

建议主程序保留以下错误码：

ERR_CONTEXT_POOL_INVALID
ERR_CONTEXT_POOL_EMPTY
ERR_CONTEXT_MEMBER_ID_DUPLICATED
ERR_P3_INPUT_INVALID

这些错误码属于宿主输入层错误，不属于 P3 的引用判断结果。

换句话说：

outside_pool / fabricated / unresolved 是 P3 业务结果
ContextPool 无效 / 输入结构错误 是宿主输入错误

二者不能混用。

X.6 宿主调用接口（冻结）

宿主对 P3 的最小调用方式固定为：

p3_result = reference_gateway.run(p3_request)

其中：

reference_gateway 是 P3 网关实例
p3_request 是宿主构造的 HostP3Request
p3_result 是 P3 标准输出 P3Result

宿主不得绕过 gateway.run()，直接拼装局部中间结果作为最终 P3 判断。

也就是说，宿主只依赖 P3 的统一出口，不依赖其内部实现细节。

X.7 宿主与主程序聚合关系

P3 的输出不是最终业务决策。
P3 的输出必须进入主程序统一聚合层，与 P2Result 一并合并成统一的 DecisionResult。

最小调用链应固定为：

p3_result = reference_gateway.run(p3_request)

decision = merge_p2_p3_to_decision(
    p2_result=p2_result,
    p3_result=p3_result,
)

这里的职责划分必须保持不变：

P3 负责引用完整性判断
聚合器负责合并 P2/P3
主程序负责生成统一 verdict 与 error code
最终业务动作由主程序执行
X.8 宿主禁止项（冻结）

以下做法全部视为错误接入。

X.8.1 禁止跳过 P3 直接放行

宿主不能因为 P2 结果“看起来不错”，就跳过 P3。
只要当前链路声称支持“基于上下文引用放行”，就必须执行 P3。

否则会出现：

语义看似支持
但引用实际越界
最终仍被错误放行
X.8.2 禁止把检索系统当作 P3 的补全器

当 P3 遇到无法归属引用时，宿主不得再去查询：

检索系统
向量数据库
外部数据库
外部 API

来“帮 P3 找答案”。

因为一旦这样做，P3 的边界就不再是当前 ContextPool，而变成了一个动态扩张的外部世界。
这会直接破坏 P3 的核心定义。

X.8.3 禁止把 P3 当成仅日志参考

P3 不是“记录一下就算了”的辅助模块。
对于 outside_pool 和 fabricated 这类硬失败结果，主程序必须在聚合层体现其阻断作用。

宿主不能：

记录日志
打个标记
然后继续正常 allow

否则 P3 将失去主线意义。

X.8.4 禁止在 P3 前做引用改写

宿主不得为了“提高通过率”而在 P3 前进行：

自动补引用
自动替换引用 ID
自动删除异常引用
自动格式修复

这些行为应归属于未来的外部纠错/重试插件，不属于主程序正常 P3 接入链路。

X.8.5 禁止将无 ContextPool 情况降级为语义判断

当没有 ContextPool 时，宿主不能说：

“那就不做引用判断了，改按语义大致判断一下。”

这会把 P3 偷偷改造成 P2。
在无 ContextPool 的情况下，应直接走宿主输入错误分支，而不是语义替代分支。

X.9 宿主最小责任边界

宿主在接入 P3 时，只承担以下责任：

构建合法的 ContextPool
提供最终待放行 llm_output
调用统一 P3 网关入口
将 P3 输出送入统一聚合层
执行主程序最终决策

宿主不负责：

替 P3 猜测引用归属
替 P3 做语义解释
替 P3 自动修复模型输出
在 P3 内部嵌入业务特例

这条边界必须严格保持，以避免主程序和 P3 相互污染。

X.10 与审计链的关系

为了保证结果可追溯，P3 参与放行链时，主程序应记录至少以下审计事件：

context_pool_built
reference_integrity_started
reference_integrity_completed
reference_integrity_failed
decision_generated

其中，P3 自身不负责审计落盘。
审计事件的记录与持久化仍属于主程序职责。

这可以保证：

P3 保持纯判断模块
主程序保持统一审计出口
整个放行链条具备复盘能力
X.11 本章结论

P3 的正确接入方式，不是把它做成一个“可有可无的检查器”，而是把它固定为：

主程序最终放行前的引用完整性网关

其接入原则可以归纳为三句话：

先有 ContextPool，再有 P3。
先做 P3，再做最终放行。
P3 只负责判断，不负责修复、不负责放行。

只要宿主违反这三条中的任意一条，P3 的设计边界就会被破坏，最终退化为一个不可靠、不可审计、不可复盘的附属模块。
