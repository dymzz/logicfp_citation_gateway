# P3 回归执行规范与结果报告模板

## 7.1 本章目标

本章用于定义 P3 benchmark 的执行方式、结果判定方式与报告输出方式，目标不是追求“跑完就算”，而是保证：

- benchmark 可重复执行；
- 回归结果可审计；
- 失败原因可定位；
- 历史版本可比较；
- 规则迭代不会因为报告口径不一致而失真。

本章的核心目标，是把 P3 的 benchmark 从“样本集合”推进为“标准化回归流程”。

---

## 7.2 适用范围

本章适用于：

- P3 benchmark 的日常执行；
- parser / resolver / evaluator 修改后的回归验证；
- 冻结版本之间的结果比较；
- benchmark 报告归档与复盘。

本章不适用于：

- P2 语义评测执行规范；
- 主程序完整集成压测；
- 行业专项插件评测流程；
- 模型质量评测报告。

---

## 7.3 回归执行目标（冻结）

P3 回归执行只回答以下问题：

1. 当前实现是否仍满足冻结状态边界；
2. 当前实现是否仍能正确解析并绑定引用；
3. 当前实现是否对 hard fail 保持稳定；
4. 当前版本相较上一版本是否出现边界漂移；
5. 失败是否集中在某一类样本或某一类路径。

P3 回归执行不回答：

- 模型整体效果提升了多少；
- 用户体验是否更好；
- 回答是否更自然；
- 语义正确率是否更高；
- 最终业务价值是否提升。

这些问题不属于 P3 回归执行的职责。

---

## 7.4 最小回归流程（冻结）

P3 benchmark 的最小执行流程固定为：

```text
加载 benchmark manifest
    ↓
按样本列表逐条加载 case
    ↓
构造 ContextPool 与 llm_output
    ↓
执行 P3 网关
    ↓
得到实际 P3Result
    ↓
与 expected 做字段比对
    ↓
记录通过 / 失败
    ↓
汇总统计
    ↓
输出标准报告

该流程中，不允许插入：

外部检索补全；
人工修正引用；
动态修改样本 expected；
模型参与解释；
任意基于当前结果的临时补救逻辑。

回归流程必须保持：

纯离线；
纯确定性；
可重复；
可归档。
7.5 回归执行输入约束（冻结）

每次回归执行，输入必须明确固定：

benchmark_version
case_set
runner_version
code_version

其中：

7.5.1 benchmark_version

表示样本集版本。
用于标识当前运行所使用的 benchmark 真值集。

7.5.2 case_set

表示本次实际执行的样本集合。
允许是全量集合，也允许是明确说明的子集，但必须被记录。

7.5.3 runner_version

表示执行器版本。
用于区分“样本真值没变，但 runner 统计口径变了”的情况。

7.5.4 code_version

表示被测实现版本。
建议记录 git commit、tag 或明确版本号。

如果这些信息缺失，则回归结果在后续复盘中将无法可靠对齐。

7.6 单条样本执行规则（冻结）

对每条 case，执行流程固定为：

读取 context_pool
读取 llm_output
构造 HostP3Request
调用 gateway.run()
获取 P3Result
与 expected 做字段比对
生成该 case 的 pass / fail 记录

单条样本执行必须完全独立。

前一条 case 的执行状态不得影响后一条 case。
因此，P3 benchmark runner 不允许：

共享跨 case 的缓存状态；
复用上条 case 的 pool；
依赖上一条执行结果做推断；
因上一条失败而修改后一条输入。

每条 case 都必须视为一个独立的确定性判定单元。

7.7 结果比对分层（冻结）

为了避免回归过于脆弱，同时又保护核心边界，本章定义三层比对规则。

7.7.1 一级必比字段（硬比较）

以下字段必须严格一致：

reference_integrity_status

这是 P3 回归中最核心的字段。
一旦该字段发生变化，就不再是“解释文字差异”，而是状态边界变化。

7.7.2 二级关键字段（强比较）

以下字段建议纳入强比较：

resolved_members
unresolved_members
outside_pool_references
triggers

这些字段用于帮助判断：

状态为什么变化；
漂移发生在 parser、resolver 还是 evaluator；
问题是单条 case 误差，还是一类规则整体漂移。
7.7.3 三级辅助字段（弱比较）

以下字段可作为弱比较或附加比较：

reasons
metrics
报告附加统计信息

这些字段如果变化，不一定代表边界失真；
但如果出现显著漂移，仍应在报告中记录，而不是直接吞掉。

7.8 失败分级（冻结）

为了避免回归结果退化为“只看 pass / fail”的粗糙口径，本章固定失败分级如下。

7.8.1 pass

核心状态与关键字段均符合预期。
可视为稳定通过。

7.8.2 soft_fail

一级核心字段未变，但二级或三级字段出现差异。
例如：

reference_integrity_status 正确；
但 triggers 与 expected 不一致；
或解释字段发生偏移。

soft_fail 不一定阻塞发布，但必须记录。

7.8.3 medium_fail

一级核心字段正确，但二级关键字段出现较大差异，说明实现可能存在局部退化。
例如：

status = partial 保持不变；
但 resolved_members / unresolved_members 显著偏移；
或 category 内多条 case 出现一致性异常。

medium_fail 通常需要人工复查，不应被当作普通噪声忽略。

7.8.4 hard_fail

一级核心字段发生错误，尤其是以下情况：

clean 被打成 unresolved
outside_pool 被降级为 partial
fabricated 被降级为 unresolved
hard fail 路径失去阻断语义

hard_fail 应视为边界回归失败，默认进入发布阻塞或强审查路径。

7.9 建议的失败判定规则

建议采用如下最小判定方式：
```python
def judge_case_result(expected, actual):
    if actual.reference_integrity_status != expected.reference_integrity_status:
        return "hard_fail"

    key_fields_changed = any([
        actual.resolved_members != expected.get("resolved_members", actual.resolved_members),
        actual.unresolved_members != expected.get("unresolved_members", actual.unresolved_members),
        actual.outside_pool_references != expected.get("outside_pool_references", actual.outside_pool_references),
    ])

    trigger_changed = actual.triggers != expected.get("triggers", actual.triggers)

    if key_fields_changed:
        return "medium_fail"

    if trigger_changed:
        return "soft_fail"

    return "pass"
```

这段伪代码的意义不是冻结具体实现细节，
而是冻结“失败分级优先于简单二元成败”的执行思想。

7.10 类别统计要求（冻结）

每次回归报告必须至少按以下维度统计：

clean
partial
unresolved
outside_pool
fabricated

报告中至少应输出：

每类总数；
每类通过数；
每类失败数；
每类失败级别分布（如可行）。

这样做的原因是：

可快速定位失败集中在哪一类；
可快速判断是否是 hard fail 类整体退化；
可避免“总体通过率好看，但关键类别崩了”的假象。
7.11 发布阻塞建议（建议冻结）

建议将以下情况默认视为 release blocking：

任意 hard_fail 出现在：
outside_pool
fabricated
任意类别出现批量状态漂移；
benchmark_version / runner_version / code_version 缺失；
回归过程中发生样本动态改写。

建议将以下情况默认视为非立即阻塞、但必须记录：

少量 soft_fail
个别可解释的 medium_fail
报告附加字段变化但核心状态未变

最终是否阻塞发布，由主程序或仓库治理规则决定；
但 P3 报告应明确给出“是否建议阻塞”的结论字段。

7.12 回归执行禁止项（冻结）

以下做法全部视为错误执行方式。

7.12.1 禁止动态修改 expected

执行过程中不得因为“当前结果不一致”而临时修改样本真值。
如果结果异常，必须先回答：

代码是否错了；
样本真值是否错了；
设计边界是否正式升级了。

没有明确理由，不允许调整 expected。

7.12.2 禁止为当前代码“洗报告”

不能因为想让结果好看，就：

只报 pass；
忽略 medium_fail；
把 hard_fail 合并成普通 fail；
把 fabricated 降级问题描述成“小偏差”。
7.12.3 禁止跳过 hard fail 类样本

为了“让报告更好看”，不能跳过：

outside_pool
fabricated

这两类正是 P3 的核心保护路径。
一旦跳过，P3 的 benchmark 将失去存在意义。

7.12.4 禁止把 soft_fail 伪装成 pass

如果 triggers、解释字段、关键附加信息发生变化，不能直接吞掉。
即使不阻塞，也应如实记录为 soft_fail 或 drift。

7.12.5 禁止无版本执行

没有 benchmark_version / runner_version / code_version 的回归结果，不应进入正式归档。
否则后续无法做版本对照。

7.13 建议的最小报告输出形式

当前阶段不需要一开始就做复杂前端展示。
建议优先支持两种报告形式：

7.13.1 控制台摘要

适合本地快速回归。
最小应包含：

总数
通过数
soft_fail 数
medium_fail 数
hard_fail 数
category 统计
简短失败清单
7.13.2 Markdown 报告

适合归档与评审。
要求：

结构清晰；
可直接提交到仓库；
可用于 PR / 设计评审记录；
可与历史版本做肉眼比较。

如果后续需要，再扩展到：

JSON 明细报告
HTML 报告
可视化 dashboard

但这些都不是当前阶段主线。

7.14 建议的最小 Markdown 报告模板
# P3 Benchmark Report

- benchmark_name: p3_reference_integrity
- benchmark_version: v1
- runner_version: v1
- code_version: <git commit>

## Summary
- total_cases: 25
- pass: 21
- soft_fail: 2
- medium_fail: 1
- hard_fail: 1

## By Category
- clean: total=5, pass=5, fail=0
- partial: total=5, pass=4, fail=1
- unresolved: total=5, pass=4, fail=1
- outside_pool: total=5, pass=5, fail=0
- fabricated: total=5, pass=3, fail=2

## Failed Cases

### p3_fabricated_001
- category: fabricated
- failure_level: hard_fail
- expected.status: fabricated
- actual.status: unresolved
- diff_summary: fabricated hard-fail path regressed to unresolved

## Conclusion
- hard_fail_detected: yes
- release_blocking: yes
- notes: fabricated category needs review

这份模板的重点不是华丽，而是：

统一；
易读；
可归档；
可比较。
7.15 推荐的报告最小字段

每份正式归档报告建议至少包含：

benchmark_name
benchmark_version
runner_version
code_version
total_cases
pass
soft_fail
medium_fail
hard_fail
by_category
failed_cases
conclusion

如果缺失这些基本字段，报告将难以用于长期版本比较。

7.16 与样本、版本控制、主程序集成的关系

本章只定义：

如何执行 benchmark；
如何判定 pass / fail；
如何输出标准报告。

本章不重新定义：

样本真值；
状态枚举；
trigger / error code；
主程序最终业务 verdict。

因此，发生冲突时应按以下顺序回看：

附录 A：术语表与冻结枚举总表
P3_05：Benchmark 与回归样本约束
P3_06：Benchmark 样本编写规范与 Case 模板
P3_08：版本演进与变更控制约束
本章
7.17 本章结论

P3 benchmark 如果只有样本而没有执行规范，就很容易退化为“有人手动跑过一次”的非工程化资产。

因此，本章冻结三条核心原则：

状态边界优先于表面通过率；
失败分级优先于简单二元成败；
报告标准化优先于临时口头结论。

P3 的回归体系只有在“样本、执行、报告”三者同时标准化后，才能真正起到边界守卫作用。
