# P3 最小接入时序与宿主调用约束

## 4.1 本章目标

本章不再讨论 P3 的能力边界本身，而是明确：

1. 宿主系统应在什么位置接入 P3；
2. P3 执行前必须满足哪些前置条件；
3. 哪些错误属于宿主输入错误，而不是 P3 业务结果；
4. 宿主对 P3 的最小调用方式应该如何固定；
5. 哪些接入方式属于错误实现；
6. P3 返回后，如何进入主程序统一决策链。

本章的目标，是把 P3 从“一个存在于设计图中的能力模块”，收敛为“主程序最终放行前的固定节点”。

---

## 4.2 接入定位（冻结）

P3 的正式定位是：

**主程序最终放行前的引用完整性校验节点**

这意味着：

- P3 不是离线审计器；
- P3 不是事后分析器；
- P3 不是可选日志插件；
- P3 不是仅供人工排查时参考的辅助模块。

P3 必须位于：

**待放行输出**  
与  
**主程序统一 DecisionResult 生成**

之间。

换句话说：

只要当前链路宣称支持“基于上下文引用的放行约束”，  
那么 P3 就必须成为主程序主链路中的固定步骤，而不是旁路组件。

---

## 4.3 最小时序（冻结）

宿主侧最小时序固定为：

```text
Host 构建 ContextPool
    ↓
Host 获取最终待放行 llm_output
    ↓
Host 生成 HostP3Request
    ↓
P3 Parser 解析引用
    ↓
P3 Resolver 绑定 ContextPool
    ↓
P3 Evaluator 输出 P3Result
    ↓
主程序聚合 P2Result + P3Result
    ↓
生成统一 DecisionResult
    ↓
主程序决定 allow / warn / fallback / block
```
本时序中，必须写死以下原则：

先有 ContextPool，再有 P3；
先做 P3，再做最终 decision；
P3 不直接放行，也不直接修复，只负责判断。

其中最关键的一条是：

P3 必须在主程序最终放行前执行，不能后置为审计补充项。

## 4.4 宿主前置条件（冻结）

宿主在调用 P3 前，必须满足以下条件。

### 4.4.1 ContextPool 已构建完成

P3 的唯一合法引用边界来源是 ContextPool。
如果 ContextPool 尚未构建完成，则 P3 无法定义：

什么属于池内引用；
什么属于池外引用；
什么是合法归属边界。

因此宿主必须保证：

当前轮可用上下文已经收口；
当前轮允许引用的成员集合已经确定；
ContextPool.members 是本轮放行中唯一合法的引用成员集合。
### 4.4.2 member_id 唯一且稳定

宿主必须保证每个 ContextMember.member_id：

非空；
唯一；
在本轮调用中稳定不变。

如果存在重复 member_id，则引用归属关系会失去确定性。
这种情况不应交给 P3“自行猜测”，而应视为宿主输入错误。

### 4.4.3 llm_output 必须是最终待放行文本

送入 P3 的 llm_output 必须是：

主程序准备交付给用户或下游系统的最终文本

不允许把以下对象送入 P3：

中间态文本；
草稿文本；
尚未收口的增量文本；
后续还会被二次改写的半成品输出。

否则会导致：

P3 校验对象与最终实际交付对象不一致；
审计链断裂；
DecisionResult 无法追溯到真实输出。
### 4.4.4 宿主不得在 P3 前修改引用

宿主在调用 P3 前，不得对引用做如下处理：

自动补引用；
自动替换引用 ID；
自动重写引用格式；
自动删除异常引用；
自动做“容错修复”。

P3 必须面对模型原始的最终待放行输出，而不是被宿主预处理后的“清洗版输出”。

否则，P3 校验的就不再是模型真实产物，
而是宿主改写后的衍生版本。

## 4.5 宿主输入错误处理约束

当宿主前置条件不满足时，不应进入正常 P3 判断流程。
这些问题应由主程序直接进入输入错误分支，而不是伪装成 P3 业务结果。

建议宿主输入错误统一保留以下错误码：

ERR_CONTEXT_POOL_INVALID
ERR_CONTEXT_POOL_EMPTY
ERR_CONTEXT_MEMBER_ID_DUPLICATED
ERR_P3_INPUT_INVALID

它们分别表示：

### 4.5.1 ERR_CONTEXT_POOL_INVALID

ContextPool 结构不合法，无法作为 P3 的边界输入。

### 4.5.2 ERR_CONTEXT_POOL_EMPTY

ContextPool 为空，或缺失成员集合，导致池内 / 池外边界不可定义。

### 4.5.3 ERR_CONTEXT_MEMBER_ID_DUPLICATED

ContextPool.members 中存在重复 member_id，导致归属关系失去唯一性。

### 4.5.4 ERR_P3_INPUT_INVALID

宿主传入的 HostP3Request 不满足最小结构约束，例如：

缺少 request_id
缺少 llm_output
缺少 context_pool
缺少 decision_type
缺少 action_risk

必须强调：

outside_pool / fabricated / unresolved / partial / clean 是 P3 业务结果
ERR_CONTEXT_POOL_INVALID / EMPTY / DUPLICATED / ERR_P3_INPUT_INVALID 是 宿主输入错误

这两类问题不能混用。

## 4.6 HostP3Request 最小输入契约

宿主传给 P3 的统一输入对象固定为：

```python
from dataclasses import dataclass, field
from typing import Any

@dataclass
class ContextMember:
    member_id: str
    source_id: str
    content: str
    title: str | None = None
    metadata: dict[str, Any] = field(default_factory=dict)

@dataclass
class ContextPool:
    pool_id: str
    members: list[ContextMember]
    retrieval_meta: dict[str, Any] = field(default_factory=dict)

@dataclass
class HostP3Request:
    request_id: str
    llm_output: str
    context_pool: ContextPool
    decision_type: str
    action_risk: str
    metadata: dict[str, Any] | None = None
```
说明：

request_id：本次调用的唯一请求标识；
llm_output：最终待放行文本；
context_pool：当前轮合法引用边界；
decision_type：决策类型；
action_risk：动作风险等级；
metadata：扩展信息，不参与核心状态判定。

宿主不得绕过该统一入口，
以零散参数拼接或内部中间对象直接调用 P3 的局部组件。

## 4.7 宿主统一调用接口（冻结）

宿主对 P3 的最小调用方式固定为：

p3_result = reference_gateway.run(p3_request)

P3 返回后，宿主必须进入统一聚合层：

decision = merge_p2_p3_to_decision(
    p2_result=p2_result,
    p3_result=p3_result,
)

因此，宿主侧最小固定动作只有两个：

调 P3；
合并决策。

后续无论 P3 内部如何重构：

parser 如何拆分；
resolver 如何演进；
evaluator 如何优化；
metrics 如何补充；

宿主都不应依赖内部细节，只依赖：

reference_gateway.run(...)
merge_p2_p3_to_decision(...)
## 4.8 与主程序聚合链的关系

P3 的输出不是最终业务决策。
P3 的输出必须进入主程序统一聚合层，与 P2Result 一起合并为 DecisionResult。

职责划分固定如下：

P3：负责引用完整性判断；
P2：负责语义支持判断；
聚合器：负责收敛为统一 DecisionResult；
主程序：负责执行最终 allow / warn / fallback / block；
P4：如有，负责外部行业专项、人工审核、自动纠错、组织流程等扩展能力。

因此，宿主对 P3 的正确理解不是：

“P3 自己决定放不放。”

而是：

“P3 给出引用完整性判断，主程序统一消费并作最终决策。”

## 4.9 宿主禁止项（冻结）

以下做法全部视为错误接入。

### 4.9.1 禁止跳过 P3 直接按 P2 放行

只要当前链路声称支持基于上下文引用的放行约束，
宿主就不能因为：

P2 看起来支持；
语义大致没问题；
输出文风很像正确答案；

而跳过 P3。

否则会出现：

语义看似支持；
实际引用已经越界；
最终仍被错误放行。
### 4.9.2 禁止把检索系统当作 P3 的补全器

当 P3 遇到无法归属的引用时，宿主不得再调用：

检索系统；
向量数据库；
外部数据库；
外部 API；
任意外部知识源；

来“替 P3 找答案”。

一旦这样做，P3 的合法边界就不再是当前 ContextPool，
而会退化成一个不断动态扩张的外部世界。
这会直接破坏 P3 的核心定义。

### 4.9.3 禁止把 P3 结果当成仅日志参考

P3 不是“记一下日志就算了”的模块。
对于以下结果：

outside_pool
fabricated

主程序必须在聚合层体现它们的阻断效力。

宿主不能：

记录日志；
打个风险标签；
然后仍然继续 allow。

否则 P3 会失去主线意义。

### 4.9.4 禁止在 P3 前做引用改写

宿主不得为了提高通过率，在 P3 前执行：

自动补引用；
自动替换引用 ID；
自动删掉异常引用；
自动格式修正。

这些行为如果未来需要存在，也应属于外部纠错 / 重试插件范畴，
不属于主程序对 P3 的正常接入链路。

### 4.9.5 禁止在 outside_pool / fabricated 时继续 allow

只要 P3 已经给出：

outside_pool
fabricated

宿主就不得继续允许主链路正常放行。

这类状态属于硬失败路径，
不允许被“人工感觉差不多”“P2 似乎支持”“业务方想先放出去”所覆盖。

### 4.9.6 禁止无 ContextPool 时降级为语义判断

在没有 ContextPool 的情况下，宿主不能说：

“那就先不做引用判断了，改按语义大概看一下。”

这会把 P3 偷偷改造成 P2。
无 ContextPool 的情况应直接进入宿主输入错误分支，而不是语义替代分支。

## 4.10 宿主最小责任边界

宿主在接入 P3 时，只承担以下责任：

构建合法的 ContextPool；
提供最终待放行 llm_output；
构造标准 HostP3Request；
调用统一 reference_gateway.run()；
将 P3Result 送入统一聚合层；
执行主程序最终决策。

宿主不负责：

替 P3 猜测引用归属；
替 P3 做语义解释；
替 P3 自动修复模型输出；
在 P3 内部嵌入业务特例；
替 P3 调外部系统补全证据。

这条边界必须保持严格稳定，
否则主程序与 P3 会相互污染。

## 4.11 审计链要求

为了保证接入链可追溯，P3 进入放行主链后，主程序应至少记录以下审计事件：

context_pool_built
reference_integrity_started
reference_integrity_completed
reference_integrity_failed
decision_generated

建议每条事件至少带以下最小字段：

request_id
trace_id
pool_id
p3_status
p2_status（如已存在）
decision_verdict
trigger_codes
error_codes
timestamp

其中必须明确：

P3 自身不负责审计落盘；
审计事件记录与持久化属于主程序职责；
P3 只输出判断结果，不承担日志系统实现。

这样可以保证：

P3 保持纯判断模块；
主程序保持统一审计出口；
整条放行链具备复盘能力。
## 4.12 推荐的宿主实现骨架

建议宿主接入骨架最小固定为：
```python
def run_reference_integrity_stage(
    *,
    request_id: str,
    llm_output: str,
    context_pool,
    decision_type: str,
    action_risk: str,
    p2_result,
    reference_gateway,
):
    validate_context_pool(context_pool)
    validate_p3_input(
        request_id=request_id,
        llm_output=llm_output,
        context_pool=context_pool,
        decision_type=decision_type,
        action_risk=action_risk,
    )

    p3_request = HostP3Request(
        request_id=request_id,
        llm_output=llm_output,
        context_pool=context_pool,
        decision_type=decision_type,
        action_risk=action_risk,
        metadata=None,
    )

    p3_result = reference_gateway.run(p3_request)

    decision = merge_p2_p3_to_decision(
        p2_result=p2_result,
        p3_result=p3_result,
    )
    return decision
```

该骨架的意义不是限定具体代码结构，
而是固定主程序接入 P3 时必须遵守的流程顺序与责任边界。

## 4.13 本章结论

P3 的正确接入方式，不是把它做成一个“有就更好、没有也能跑”的检查器，
而是把它固定为：

主程序最终放行前的引用完整性网关

本章应冻结三条原则：

先有 ContextPool，再有 P3；
先做 P3，再做最终放行；
P3 只负责判断，不负责修复、不负责放行。

只要宿主违反这三条中的任意一条，
P3 的设计边界就会被破坏，最终退化为一个：

不可靠；
不可审计；
不可复盘；
不可稳定回归；

的附属模块。
